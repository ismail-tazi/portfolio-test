The challenge

Raw sales data sitting in a warehouse isn't analytics-ready on its own — without a clear modeling layer, every report ends up rebuilding the same transformations from scratch, inconsistently.

The approach

  • Loaded the raw retail sales data into Snowflake.
  • Structured the transformation logic into a layered dbt project: raw → staging → intermediate → marts.
  • Added dbt tests (not_null, unique, relationships) to catch data-quality issues before they reach a dashboard.
  • Built a couple of incremental models and documented every model's grain and purpose with dbt docs.

The result

The warehouse now has a single, tested, documented source of truth for sales data — any new report builds on the marts layer instead of re-deriving the same numbers a different way.

  • 3-layer dbt project (staging, intermediate, marts)
  • Automated tests catch data-quality issues before dashboards do
  • Incremental models keep transformations fast as data grows

What I'd do differently

Deciding the right grain for each mart table was harder than writing the SQL itself. I'd spend more time upfront sketching the intended marts on paper before writing a single model, rather than discovering the right grain through trial and error.
