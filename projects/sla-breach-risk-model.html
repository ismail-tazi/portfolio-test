The challenge

Knowing that port-out requests breach SLA more often than other ticket types (from the earlier backoffice analysis) is useful, but it doesn't say which specific open tickets are about to breach today. Turning that insight into a per-ticket risk score needed a predictive model, not just a descriptive one.

The approach

  • Reused the backoffice ticket data from the descriptive SLA analysis and engineered additional features (ticket age, request type, agent's current workload, time of day submitted).
  • Trained a logistic regression model as an interpretable baseline.
  • Trained a random forest model and compared it against the baseline.
  • Evaluated both on precision and recall rather than accuracy alone, since SLA breaches are a minority class where accuracy alone is misleading.
  • Translated the top features into a plain prioritization rule instead of leaving them as model coefficients.

The result

The model flags at-risk tickets while they're still open, and the feature importance translates directly into a prioritization rule for the backoffice team: port-out tickets assigned to already-overloaded agents late in the day are far more likely to breach SLA.

  • Logistic regression baseline vs. random forest comparison
  • Evaluated on precision/recall, not just accuracy
  • Feature importance translated into an agent-prioritization rule

What I'd do differently

This project taught me that a good model is only half the deliverable — translating 'feature X has coefficient Y' into a prioritization rule a backoffice supervisor can act on took as much effort as the modeling itself. I'd start with that operational translation next time, and let it guide which features I engineer.
