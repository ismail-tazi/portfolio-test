The challenge

A telecom reseller's backoffice team processes a constant stream of port-in, port-out, deactivation, and provisioning requests, but without visibility into ticket volume, turnaround time, and SLA compliance by agent and request type, backlogs build up invisibly until customers start complaining.

The approach

  • Queried and segmented backoffice ticket data in SQL by request type (port-in, port-out, deactivation, provisioning projects, listing updates), agent, and status.
  • Calculated turnaround time and SLA compliance for each request type to find where delays concentrated.
  • Built an agent-level workload view to see ticket volume and backlog per agent, not just per request type.
  • Delivered the findings as a Power BI report plus a one-page executive summary in French, matching the format used internally by the reseller's operations team.

The result

Port-out requests turned out to breach SLA far more often than port-in or deactivation requests — a specific, actionable finding instead of a vague sense that 'the backoffice is behind.'

  • Tickets broken down by 5 request types
  • Agent-level workload and backlog view built from raw ticket data
  • 1-page executive summary delivered in French

What I'd do differently

The SQL segmentation by request type was straightforward — the harder part was agreeing on what actually counted as an SLA breach with the operations team, since definitions varied informally by request type before this project. I'd nail down those definitions with stakeholders before writing a single query next time.
