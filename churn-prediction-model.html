The challenge

Knowing that month-to-month customers churn more (from the earlier descriptive analysis) is useful, but it doesn't say which specific customers are about to leave. Turning that insight into a per-customer risk score needed a predictive model, not just a descriptive one.

The approach

  • Reused the churn dataset from the descriptive analysis and engineered additional features (tenure buckets, support-ticket counts, contract type).
  • Trained a logistic regression model as an interpretable baseline.
  • Trained a random forest model and compared it against the baseline.
  • Evaluated both on precision and recall rather than accuracy alone, since churn is an imbalanced problem where accuracy is misleading.
  • Translated the top features into plain business language instead of leaving them as model coefficients.

The result

The model flags high-risk customers before they cancel, and the feature importance translates directly into a retention talking point: customers on month-to-month contracts with high support-ticket counts are far more likely to churn.

  • Logistic regression baseline vs. random forest comparison
  • Evaluated on precision/recall, not just accuracy
  • Feature importance explained in plain business language

What I'd do differently

This project taught me that a good model is only half the deliverable — translating 'feature X has coefficient Y' into something a retention manager can act on took as much effort as the modeling itself. I'd start with the business translation next time, and let it guide which features I engineer.
